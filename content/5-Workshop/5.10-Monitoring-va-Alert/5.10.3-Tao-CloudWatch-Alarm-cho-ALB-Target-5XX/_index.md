### 5.10.3. Create an Alarm for ALB Target HTTP 5XX Errors

This CloudWatch alarm monitors HTTP 5XX responses returned by targets behind the Application Load Balancer. When the backend produces a server error, the alarm enters the **In alarm** state and sends a notification through the SNS topic.

#### Step 1: Select the HTTPCode_Target_5XX_Count metric

Open **CloudWatch → Alarms → All alarms → Create alarm → Select metric**. Navigate to **ApplicationELB → Per AppELB Metrics**, search for `HTTPCode_Target_5XX_Count`, and select the `recruit-alb` load balancer.

![Select the ALB Target 5XX error metric](</images/5-Workshop/5.10-Monitoring-va-Alert/5.10.3-Tao-CloudWatch-Alarm-cho-ALB-Target-5XX/chon-metric-alb-target-5xx.jpg>)

This metric counts 5XX responses generated by targets, such as errors from the backend application. It differs from `HTTPCode_ELB_5XX_Count`, which counts errors generated by the load balancer itself. Choose **Select metric** to continue.

#### Step 2: Configure the statistic and error threshold

Configure the metric as follows:

- **Namespace:** `AWS/ApplicationELB`.
- **Metric name:** `HTTPCode_Target_5XX_Count`.
- **Statistic:** `Sum`, to total the errors in each period.
- **Period:** `5 minutes`.
- **Threshold type:** `Static`.
- **Condition:** `Greater/Equal` with a threshold of `1`.

![Configure the ALB Target 5XX threshold](</images/5-Workshop/5.10-Monitoring-va-Alert/5.10.3-Tao-CloudWatch-Alarm-cho-ALB-Target-5XX/cau-hinh-nguong-target-5xx.jpg>)

This condition is met when at least one target returns a 5XX response within five minutes. The sensitive threshold is suitable for the workshop and low-traffic systems. In production, a higher threshold or an error-rate alarm can reduce noise from isolated failures.

#### Step 3: Send notifications to the SNS topic

On **Configure actions**, select **In alarm**, then choose **Select an existing SNS topic** and select `recruitpro-alert-topic`.

![Select the SNS topic for Target 5XX notifications](</images/5-Workshop/5.10-Monitoring-va-Alert/5.10.3-Tao-CloudWatch-Alarm-cho-ALB-Target-5XX/chon-sns-topic.jpg>)

The email endpoint shown below the topic confirms that a recipient is subscribed. When the threshold is breached, CloudWatch publishes the event to SNS, which forwards the notification by email. Lambda, Auto Scaling, EC2, and Systems Manager actions are not required here.

#### Step 4: Name the alarm

In **Add alarm details**, enter a name that identifies the resource and error type, such as `RecruitPro-ALB-Target-5XX`.

![Name the ALB Target 5XX CloudWatch alarm](</images/5-Workshop/5.10-Monitoring-va-Alert/5.10.3-Tao-CloudWatch-Alarm-cho-ALB-Target-5XX/dat-ten-alarm.jpg>)

The image displays **The alarm name already exists**, which means an alarm with this name already exists in the same Region. CloudWatch alarm names must be unique. Use a unique name such as `RecruitPro-ALB-Target-5XX-v2` for a new alarm, or edit the existing alarm if it already represents the intended configuration.

#### Step 5: Review and create the alarm

On **Preview and create**, verify the metric, condition, action, and alarm name.

![Review the Target 5XX CloudWatch alarm](</images/5-Workshop/5.10-Monitoring-va-Alert/5.10.3-Tao-CloudWatch-Alarm-cho-ALB-Target-5XX/xem-lai-va-tao-alarm.jpg>)

The Review page confirms `HTTPCode_Target_5XX_Count`, the `Sum` statistic, a five-minute period, a threshold greater than or equal to 1, and notifications to `recruitpro-alert-topic`. After ensuring that the alarm name is unique, choose **Create alarm**. The alarm may initially show **Insufficient data**, then change to **OK** when no errors occur or **In alarm** when a target returns a 5XX response.
